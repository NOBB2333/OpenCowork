# Electron 进程模型 / Electron Process Model

OpenCowork 的 Electron 进程模型已经从传统“三层 UI 应用”演进为“主进程编排 + 渲染进程 Agent Runtime + 预加载桥接 + sidecar 扩展运行时”的组合架构。

## 主进程 / Main Process

**入口**：`src/main/index.ts`

主进程负责所有系统级能力与运行时编排，当前主要包括：

- 创建主窗口与通知窗口环境
- 管理托盘、窗口显隐、窗口状态同步
- 注册完整 IPC 处理器集合
- 初始化 `ChannelManager`、`McpManager`、`SidecarManager`
- 加载持久化 cron jobs 并恢复后台运行能力
- 配置系统代理、Chromium 缓存、渲染堆内存上限
- 记录崩溃、无响应、加载失败等诊断事件

## 渲染进程 / Renderer Process

**入口**：`src/renderer/src/main.tsx`

渲染进程运行 React 19 UI，并承载核心 Agent 交互运行时：

- 渲染 `App` 主界面与 `NotifyWindow`
- 维护 Zustand store 网络
- 执行 Agent loop、子代理、团队协作与审批流
- 注册本地工具、MCP 工具、应用插件工具
- 直接调用 LLM Provider API 并处理流式输出

### 窗口装配

```typescript
const isNotifyWindow = window.location.hash.startsWith('#notify')
createRoot(document.getElementById('root')!).render(isNotifyWindow ? <NotifyWindow /> : <App />)
```

### 为什么 Agent Runtime 放在渲染进程

- 更贴近 UI 状态与流式消息展示
- 工具调用、审批弹层、计划/任务面板可以即时联动
- Provider 请求无需经过主进程转发，降低链路复杂度

## 预加载脚本 / Preload Script

**入口**：`src/preload/index.ts`

预加载脚本只暴露经过筛选的桥接能力。除 Electron Toolkit 标准桥接外，还额外开放：

- 图像下载 / Base64 抓取 / 剪贴板写入
- Team Runtime 创建、消息附加、快照读取
- 独立 Team Worker 拉起与停止

```typescript
contextBridge.exposeInMainWorld('electron', electronAPI)
contextBridge.exposeInMainWorld('api', api)
```

这里的 `api` 不是任意 Node.js 暴露面，而是明确列举的受控 IPC 封装。

## Sidecar 进程 / Sidecar Process

**入口**：`src/dotnet/OpenCowork.Agent/Program.cs`

新增的 sidecar 不是 Electron 官方进程层的一部分，但它已经是实际架构中的第四个运行时：

- 由主进程通过 `SidecarManager` 启停与保活
- 使用 JSON-RPC 风格协议双向通信
- 可向渲染进程请求审批或工具执行结果
- 在开发态支持自动 `dotnet publish`
- 用于承载更强隔离、更稳定或跨平台一致的 Agent 能力

## 安全与边界 / Security Boundaries

- 渲染进程不直接访问 Node.js 系统接口
- 主进程集中处理文件、数据库、Shell、截图、输入控制、SSH 等敏感能力
- 预加载脚本只暴露白名单 API
- sidecar 通过主进程中转，不直接接管 Electron UI
- IPC / bridge 的通道按领域拆分，便于审计与渐进扩展
