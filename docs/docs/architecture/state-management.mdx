# 状态管理 / State Management

OpenCowork 使用 Zustand + Immer 组织渲染进程状态，但当前已经不是“少数几个 UI store”，而是一套围绕 Agent、项目、团队、平台与运行时编排的状态网络。所有 store 位于 `src/renderer/src/stores/`。

## 核心 Store / Core Stores

| Store | 职责 |
| --- | --- |
| `chat-store` | 项目、会话、消息、会话切换与数据库同步 |
| `agent-store` | Agent 主循环、工具调用、审批、子代理、后台进程输出、使用量聚合 |
| `task-store` | 当前 session 下的任务管理与步骤面板 |
| `plan-store` | Plan Mode 产物与计划状态 |
| `team-store` | Agent 团队、成员、后台 teammate 运行时 |
| `provider-store` | 模型提供商、模型列表与认证配置 |
| `settings-store` | 全局设置、默认目录、代理等偏好 |
| `app-plugin-store` | 应用插件配置与连接状态 |
| `channel-store` | 消息平台连接、消息路由与自动回复状态 |
| `mcp-store` | MCP Server 注册、连接与工具可用性 |
| `cron-store` | 定时任务、运行历史与调度状态 |
| `ssh-store` | SSH 连接与远程上下文 |
| `resources-store` | agents / prompts / commands / skills 等资源清单 |
| `notify-store` / `ui-store` | 通知窗口与界面状态 |

## Store 模式 / Store Pattern

多数复杂 store 采用 `create + immer + persist` 组合，并根据数据类型选择不同存储后端：

```typescript
create<State>()(
  persist(
    immer((set, get) => ({
      // state + actions
    })),
    {
      name: 'agent-store',
      storage: createJSONStorage(() => ipcStorage)
    }
  )
)
```

其中 `agent-store`、`settings-store`、`team-store` 会使用 `ipcStorage`，把持久化落到 Electron 可控存储层，而不是浏览器原生 `localStorage`。

## 分层状态原则 / State Boundaries

- **会话与业务实体**：优先由 `chat-store`、`task-store`、`plan-store` 与 SQLite 对齐。
- **运行时瞬时状态**：由 `agent-store` 承载，如流式文本、工具调用、审批、sub-agent transcript。
- **系统配置**：由 `settings-store`、`provider-store`、`app-plugin-store` 管理。
- **外部连接状态**：由 `channel-store`、`mcp-store`、`ssh-store` 管理。
- **界面状态**：由 `ui-store`、`notify-store` 管理。

## 持久化策略 / Persistence Strategy

| Store | 持久化方式 |
| --- | --- |
| `chat-store` | SQLite（通过 IPC 调 DAO） |
| `task-store` / `plan-store` / `cron-store` | SQLite（主进程 DB） |
| `agent-store` | `persist + ipcStorage` |
| `settings-store` | `persist + ipcStorage` |
| `team-store` | `persist + ipcStorage` |
| `provider-store` / `app-plugin-store` | 配置存储层（`configStorage`） |
| `ui-store` / `notify-store` | 内存或轻量本地持久化 |

## 当前演进特点 / Current Evolution

相较早期版本，状态管理有几个明显变化：

- 已从单 session 聊天扩展为 **project -> session** 工作流。
- `agent-store` 变成高复杂度运行时 store，负责工具执行轨迹与子代理 transcript 裁剪。
- 团队协作、计划模式、cron、SSH、Wiki 等能力各自拥有独立状态域，避免一个超级 store 承担全部职责。
- 持久化策略按“配置 / 运行时 / 业务实体”拆分，而不是统一塞进浏览器存储。
